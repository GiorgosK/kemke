# OAuth Tests For GSIS From `https://www.intra-kemke.gov.gr`

## Simple flow explanation

This OAuth flow has 3 actors:

1. KEMKE site: `https://www.intra-kemke.gov.gr`
2. GSIS authorize/token server: `https://test.gsis.gr`
3. The human user in the browser

The flow in simple terms is:

1. The user opens KEMKE login:
   `https://www.intra-kemke.gov.gr/auth/gsis-pa/login`
2. KEMKE creates a redirect to GSIS authorize:
   `https://test.gsis.gr/oauth2servergov/oauth/authorize?...`
3. In that redirect, KEMKE tells GSIS:
   - which application is connecting: `client_id`
   - where GSIS must send the browser back after login: `redirect_uri`
4. The user logs in on the GSIS page with personal username/password.
5. If GSIS accepts the user login, GSIS redirects the browser back to the exact `redirect_uri` that KEMKE originally sent.
6. That callback URL contains a temporary `code`.
7. KEMKE server then calls the GSIS token endpoint with:
   - `client_id`
   - `client_secret`
   - `code`
8. GSIS returns an `access_token`.
9. KEMKE calls GSIS `userinfo` with that token and logs the user in locally.

## Very important clarification about the callback URL

The callback path

```text
/oauth2-client/kemke_gsis_pa/code
```

did not come out of nowhere.

KEMKE itself sends this callback URL to GSIS as the `redirect_uri`.

That is configured in [web/sites/default/settings.local.php.live](/home/giorgos/share/gwebsites/kemke/web/sites/default/settings.local.php.live#L111):

```php
'redirect_uri' => match ($gsis_pa_environment) {
  'mock' => $base_url . '/oauth2-client/kemke_gsis_pa/code',
  'live' => $gsis_pa_live_redirect_uri,
  default => $gsis_pa_test_redirect_uri,
},
```

and on live:

```php
$gsis_pa_live_redirect_uri = 'https://www.intra-kemke.gov.gr/oauth2-client/kemke_gsis_pa/code';
$gsis_pa_test_redirect_uri = getenv('GSIS_PA_TEST_REDIRECT_URI') ?: 'https://www.intra-kemke.gov.gr/oauth2-client/kemke_gsis_pa/code';
```

So:

- GSIS is not inventing the callback URL on its own
- KEMKE tells GSIS to return to `/oauth2-client/kemke_gsis_pa/code`
- GSIS then correctly returns to that exact URL
- if that URL 404s on KEMKE, that is our application/routing problem

In short:

- authorize redirect URL is generated by KEMKE
- callback URL is chosen by KEMKE and sent to GSIS as `redirect_uri`
- GSIS is only following instructions when it redirects back there

This document keeps only the tests that still matter for the current symptom:

- the browser already reaches `https://test.gsis.gr/oauth2servergov/oauth/authorize`
- GSIS already shows the login form
- after entering username/password, GSIS returns:

`Αποτυχία στην αυθεντικοποίηση του χρήστη. Η σύνδεση σας δεν ήταν δυνατή`

That means plain website reachability is already proven. The remaining questions are:

1. Is KEMKE sending the correct OAuth client configuration?
2. Does GSIS accept the browser login and redirect back correctly?
3. Can the new server exchange the authorization code at the token endpoint?
4. Does the new server reach the GSIS `userinfo` endpoint?
5. Did GSIS whitelist the new outbound IP of the new server?

## Test 2: Prove the login entrypoint generates the correct redirect

Run from anywhere that can reach the public KEMKE site:

```bash
curl -k -I -s 'https://www.intra-kemke.gov.gr/auth/gsis-pa/login' | sed -n '1,20p'
```

You want to see a `Location:` header pointing to GSIS authorize, with:

- `client_id=THVRZH42954`
- `redirect_uri=https%3A%2F%2Fwww.intra-kemke.gov.gr%2Foauth2-client%2Fkemke_gsis_pa%2Fcode`
- `scope=read`
- `response_type=code`

If those values are wrong, the issue is in our configuration.

## Test 3: Prove whether the failure happens before KEMKE receives the callback

On the live server, watch the GSIS OAuth log while doing one fresh login attempt in the browser:

```bash
tail -f web/sites/default/files/private/gsis-pa/oauth-calls.log
```

If that path does not exist, locate it with:

```bash
find web -path '*gsis-pa/oauth-calls.log' -print
```

Then:

1. Open `https://www.intra-kemke.gov.gr/auth/gsis-pa/login`
2. Login on the GSIS page with the test user
3. Wait for the failure

Interpretation:

- If the log shows only `login_start` and `authorize_redirect`, then Drupal successfully sent the browser to GSIS, but GSIS never redirected back to KEMKE.
- If there is no callback at all, the failure is before our server receives the authorization code.
- In that case the likely causes are on the GSIS side, or a registration mismatch such as redirect URI / client registration / user authorization.

## Test 4: Prove whether the new server can talk to the token endpoint

This test matters because the old environment was changed and the new server may need a new whitelist entry.

Run this from the new live server:

```bash
curl -sS -D - \
  -X POST 'https://test.gsis.gr/oauth2servergov/oauth/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'client_id=THVRZH42954' \
  --data-urlencode 'client_secret=Kemke2025!!' \
  --data-urlencode 'grant_type=authorization_code' \
  --data-urlencode 'code=dummy-invalid-code-for-connectivity-test' \
  --data-urlencode 'redirect_uri=https://www.intra-kemke.gov.gr/oauth2-client/kemke_gsis_pa/code' \
  --data-urlencode 'scope=read'
```

What this proves:

- If you get an HTTP response from GSIS such as `400` with OAuth JSON, the new server can reach the token endpoint.
- If you get timeout, TLS error, or connect failure, then the new server cannot reach their token endpoint and the problem is network / firewall / whitelist.

What response is useful:

- `invalid_grant` is acceptable for this test because the code is intentionally fake.
- `invalid_client` would suggest client ID / client secret / client registration mismatch.

This is the most useful non-browser `curl` proof for support.

## Test 5: Full token exchange with a real authorization code

This is the best proof if GSIS login succeeds far enough to issue a `code`.

To do it:

1. Start one fresh browser login from `https://www.intra-kemke.gov.gr/auth/gsis-pa/login`
2. If GSIS redirects back to KEMKE, capture the callback URL
3. Copy the real `code` and `state`
4. Run:

```bash
CODE='PASTE_REAL_CODE_HERE'
STATE='PASTE_REAL_STATE_HERE'

curl -sS -D - \
  -X POST 'https://test.gsis.gr/oauth2servergov/oauth/token' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data-urlencode 'client_id=THVRZH42954' \
  --data-urlencode 'client_secret=Kemke2025!!' \
  --data-urlencode 'grant_type=authorization_code' \
  --data-urlencode "code=$CODE" \
  --data-urlencode "state=$STATE" \
  --data-urlencode 'redirect_uri=https://www.intra-kemke.gov.gr/oauth2-client/kemke_gsis_pa/code' \
  --data-urlencode 'scope=read'
```

Interpretation:

- `200 OK` with an access token means client credentials, redirect URI, and token connectivity are all correct.
- `invalid_grant` usually means expired/used/wrong code, or GSIS never actually completed the browser-side login.
- `invalid_client` points to client credential or GSIS registration mismatch.
- timeout / connection error points to network or whitelist.

## Test 6: Prove userinfo works from the new server

## My current view

Given your symptom, the weakest explanation is "wrong password in Drupal", because Drupal does not store a human GSIS password for this flow.

The stronger possibilities are:

- GSIS user login is failing on their side before redirect
- redirect URI or client registration on their side still points to the old host
- the new server outbound IP is not whitelisted for token / userinfo calls

So the most important tests are Test 3, Test 4, and Test 7.
